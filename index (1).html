WEEK 03–04 · Class 01

Domain Spine — Dependability

AvailabilityReliabilitySafetySecurityResilience

My Evidence (V2) — STC Pay Dependability Map

System Profile. STC Pay is a critical FinTech wallet serving millions of users across Saudi Arabia for daily transactions, international remittances, and salary transfers. Any service degradation translates directly into financial loss and public trust collapse.

Required Dependability Attributes
Availability
Must be operational 24/7, especially during peak windows like the 27th of each Gregorian month (salary day).
Security
Protection against unauthorized access, account takeover, and large-scale data breach.
Reliability
Zero tolerance for transaction errors, lost transfers, or ledger miscalculation.
Identified Risks
  • Third-party API failures (banking gateways, SADAD, international remittance partners).
  • DDoS attacks causing service unavailability at peak demand windows.
  • Database transaction deadlocks under high concurrency.
Dependability Strategy

Implement Fault Tolerance through redundancy — Active-Active server clusters so that no single node failure produces user-visible downtime — combined with strict Database ACID properties to guarantee that no partial transaction can ever corrupt the ledger.

WEEK 05–06 · Class 02

Reliability Engineering

POFODROCOFAVAILN-Version

My Evidence (V2) — Metric Choice & Architectural Strategy

Usage Profile

STC Pay traffic is intermittent but spikes violently — salary days, festive seasons, end-of-month bills. A user who initiates a transfer expects immediate confirmation. There is no "acceptable delay" tier.

Selected Metric — POFOD

Probability of Failure on Demand is the right metric here. ROCOF and Availability measure time-based behaviour, but the user does not experience time — the user experiences a single transfer attempt. A POFOD failure is catastrophic to trust. We target POFOD < 0.0001 for core wallet operations.

! Justification. Availability would let us hide failures behind statistics ("we were up 99.9% of the time"). POFOD forces honesty: did this specific user's transfer succeed?
Proposed Strategy — N-Version Redundancy

For the core transaction engine, employ Diversity (N-Version Programming): critical ledger calculations are computed independently by two algorithms written by separate teams. A comparator commits to the database only if both versions agree. A single logical bug in one version cannot corrupt user balances.

WEEK 07–08 · Class 03

Safety & Hazard Analysis

Hazard→Accident→DamageALARPFault TreeSafety Case

My Evidence (V2) — Safety Case for STC Pay

1 · Hazard Identification

The most consequential hazard in a wallet system is the "Unauthorized Double Deduction": deducting the transfer amount twice from the sender's wallet because the client retried after a network timeout while the server had already committed.

2 · The Chain (Hazard → Accident → Damage)
  • Hazard: Idempotency key absent on the transfer endpoint.
  • Accident: User taps "Send" again after a timeout; server processes a second deduction.
  • Damage: User loses funds, regulator opens an investigation, brand trust eroded.
  • Intervention: Break the chain by issuing a client-generated UUID per intent — server rejects duplicates server-side.
3 · ALARP Strategy

Implement strict Idempotency Keys on every write endpoint. If a network timeout triggers an automatic client retry, the API recognises the identical key and returns the cached response of the original request rather than executing it twice. Risk is reduced to As Low As Reasonably Practicable.

Fig. 3.1 — Fault Tree Analysis: routes to the Double-Deduction hazard
⚖ ALARP applied: the AND gate shows that even if a user double-taps and the network times out, the absence of an idempotency key is the only remaining failure point. Closing that gate alone reduces probability to negligible.

WEEK 09–10 · Group Phase · Class 04/05

Security Blueprint — CIA Triad & Misuse Cases

ConfidentialityIntegrityAvailabilityDefense-in-Depth

My Evidence (V2) — Security Assurance Case

CIA Triad Defense
Confidentiality
End-to-end encryption of KYC data; PAN masking in UI (display only last 4 digits).
Integrity
Digital signatures on every transaction payload to prevent Man-in-the-Middle alteration.
Availability
Geo-distributed load balancers + WAF rules to mitigate DDoS at the edge before reaching app tier.
Misuse Case — Session Hijacking

An attacker captures or replays a legitimate session token to impersonate the user. Without defense-in-depth, this would grant full wallet control.

Fig. 4.1 — Misuse Case: Session Hijacking threatens Transfer; MFA mitigates
Design Rationale — Defense in Depth

Even if the attacker bypasses Layer 1 (session token), they cannot execute a transfer without Layer 2 biometric/MFA confirmation through Nafath or device-level FaceID required at the per-transaction level. Compromising one layer is insufficient.

WEEK 09–10 · Class 06

Resilience Engineering

SociotechnicalCircuit BreakerGraceful Degradation

My Evidence (V2) — Resilience by Design

Sociotechnical Resilience

STC Pay operates inside a tightly regulated environment (SAMA). Resilience is not a pure technical property — it combines automated recovery with human oversight: on-call engineers, runbooks, and regulatory disclosure procedures.

Technical Defense — Circuit Breaker Pattern

When the international remittance API (e.g. Western Union) starts failing, the circuit breaker opens. The STC Pay UI gracefully disables only that specific feature and displays a friendly message, while local transfers, SADAD bill payments, and wallet top-ups continue to function perfectly. This prevents thread-pool exhaustion and the cascading failure that would otherwise take the entire wallet down.

⟲ Key insight: resilience is not "don't fail." It is "fail in a small, contained, reversible way." The circuit breaker turns a system-wide outage into a single disabled feature.

WEEK 11–12 · Class 08

Component-Based SE & Software Reuse

CBSEReusable ComponentsGlue CodeAdapter Pattern

My Evidence (V2) — Reuse Strategy

Approach to Reuse

STC Pay relies heavily on Component-Based Software Engineering to accelerate time-to-market and ensure SAMA compliance. Rather than re-implementing identity verification or government bill payment from scratch, it integrates standardized national components.

Standardized Reusable Components
Nafath / Absher
Reused for secure, government-backed user authentication and KYC. National identity verified at source.
SADAD
Integrated for bill processing — utilities, government fees, telecom — without rebuilding the biller network.
mada Pay
Reused for card tokenisation and contactless payment at POS terminals.
Component Composition & Glue Code

To integrate the SADAD component (which historically exposes XML/SOAP) with STC Pay's modern frontend (which expects JSON/REST), we employ the Adapter Pattern as glue code. The adapter reconciles interface incompatibilities without altering either the reusable component or the consuming service.

WEEK 13–14 · Class 09 & 10

Distributed Systems Architecture

MicroservicesMulti-TierAPI GatewayMiddleware

My Evidence (V2) — Distributed Design Justified

Architectural Choice

Microservices Architecture deployed on cloud infrastructure in a multi-tier topology. Each business capability — wallet ledger, authentication, local transfer, international transfer, SADAD bills, rewards — is an independently deployable service.

Justification
  • Scalability. Only the "Local Transfer Service" nodes scale up on salary day — not the entire monolith. Resources are saved; capacity meets demand precisely where it appears.
  • Reliability. If the "Reward Points Service" crashes due to a bug, the core "Wallet Ledger Service" is unaffected. Failure is contained to a single bounded context.
  • Security. Each service runs with least-privilege credentials. A compromise of the rewards service does not yield access to the ledger.
Middleware Selection

An API Gateway acts as the single secure entry point. It handles rate limiting, SSL termination, request routing, and JWT validation — keeping cross-cutting concerns out of the services themselves.

Growth, in four pillars.

HIMMA reframes the engineer's job: not "I built it" but "I can prove it works." Below are the four pillars that shaped the way I now think about every system I touch.

↗

Market Mindset

Delivery quality, documentation discipline, and iterative refinement — not just submission.

⊙

Evidence-First

From "I know" → "I can prove." Every claim is paired with the artifact that defends it.

◐

Communication

Industry-grade clarity. If a stakeholder cannot follow the reasoning, the reasoning has failed.

▣

Consistency

Discipline over inspiration. The same care in week 1 as in week 16. Rhythm is the proof.

— What changed in how I think?
"Before HIMMA, I thought software engineering meant writing code. Now I understand it means building evidence — every design decision must be justifiable, traceable, and provable."
— My biggest V1 → V2 improvement
"My first dependability submission was a definition list. V2 maps the five attributes to STC Pay's real risks and proposes a concrete fault-tolerance strategy — the difference between learning about a concept and using it."
— Safety vs Reliability — my understanding
"A system can perfectly follow its specification and still cause harm. Safety is freedom from harm; reliability is conformance to spec. A reliable weapon is still dangerous — that insight reframed everything."
— What I carry into my career
"The burden of proof is on the engineer. In safety engineering you are guilty until proven innocent. I will never ship a critical system without a Safety Case — and I now know how to build one."
